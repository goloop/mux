## Ментальна модель

Це `net/http.ServeMux` з тонким шаром ергономіки згори. Метчинг, пріоритет патернів,
обробка методів, `HEAD` для `GET`, host-патерни, екранування шляхів і редіректи
повністю виконує стандартна бібліотека. `mux` лише:

1. приклеює префікс шляху до патернів, які ви реєструєте,
2. загортає хендлери зрізом middleware на момент реєстрації,
3. додає опційну форму хендлера, що повертає `error`.

Патерни - стандартні патерни `net/http`. Немає власного синтаксису патернів,
регулярних виразів чи зворотної маршрутизації.

## Створення роутера

```go
r := mux.New()

r := mux.New(
    mux.WithErrorHandler(myErrorHandler),
    mux.WithNotFound(myNotFound),
    mux.WithMethodNotAllowed(myMethodNotAllowed),
)
```

Опції застосовуються по черзі. `Router` є `http.Handler`.

## Реєстрація маршрутів

Метод-хелпери реєструють `METHOD path`:

```go
r.Get("/users/{id}", h)     // GET /users/{id}
r.Post("/users", h)         // POST /users
r.Put("/users/{id}", h)
r.Patch("/users/{id}", h)
r.Delete("/users/{id}", h)
r.Options("/users", h)
r.Head("/users/{id}", h)    // рідко потрібно: GET уже відповідає на HEAD
r.Method("REPORT", "/x", h) // будь-який інший метод
```

`Handle` і `HandleFunc` приймають повний патерн, що може містити метод і host:

```go
r.Handle("GET example.com/x", h)
r.HandleFunc("GET /x", h)
```

`GET`-маршрут також відповідає на `HEAD` - так каже стандартний mux. Не додавайте
власних перевірок методу.

Реєстрація двох хендлерів на один патерн викликає panic - так само, як у
`net/http`. Це навмисно: помилки розробника мають випливати одразу; у v0 ми їх не
ковтаємо і не перетворюємо на `error`.

## Параметри шляху

```go
id := r.PathValue("id")   // стандартно
id := mux.Param(r, "id")  // те саме, одна очевидна назва
```

## Групи та префікси

```go
r.Route("/api/v1", func(r *mux.Router) {
    r.Get("/users/{id}", show)   // GET /api/v1/users/{id}
    r.Route("/admin", func(r *mux.Router) {
        r.Get("/stats", stats)   // GET /api/v1/admin/stats
    })
})
```

`Group` - це `Route` без додавання префікса; існує, щоб обмежити middleware:

```go
r.Group(func(r *mux.Router) {
    r.Use(auth)
    r.Get("/me", showMe)
    r.Post("/logout", logout)
})
```

Склеювання префіксів нормалізує слеші: `/api` і `/api/` поводяться однаково, шлях
без початкового слеша його отримує, а `/{$}` (точний збіг) зберігається.

## Middleware

Middleware - це `func(http.Handler) http.Handler`.

- `Use(m...)` додає middleware до поточного роутера. Вони діють на маршрути,
  зареєстровані **після** цього, і на створені з нього суброутери.
- `With(m...)` повертає суброутер із додатковими middleware, не змінюючи батька -
  зручно для одного маршруту: `r.With(auth).Get("/me", h)`.
- `Chain(m...)` склеює middleware в одне.

Middleware фіксуються під час реєстрації маршруту, тож route-scoped middleware
можуть читати параметри шляху. Root-middleware (додані через `Use` на роутері з
`New`) виконуються також для власних відповідей `404` і `405`, тож
security-заголовки, CORS і логування покривають і помилкові відповіді. На цьому
шляху немає збіжного маршруту, тож `PathValue`/`Param` порожні; middleware, що
розраховує на збіжний маршрут, варто застосовувати поper-route. Для загальних
middleware, які мають бачити кожен запит (зокрема редіректи й `OPTIONS *`),
загорніть сам роутер:

```go
http.ListenAndServe(":8080", mux.Chain(requestID, recoverer, logger)(r))
```

Порядок: перший middleware - найзовнішніший. `Use(A)`, потім група з `Use(C)`,
потім `With(B)` на маршруті дають `A -> C -> B -> handler`.

`With` ніколи не змінює батька: middleware, додані дитині, не витікають назад.

## Хендлери, що повертають error

```go
type HandlerFunc func(http.ResponseWriter, *http.Request) error
```

Реєстрація через `E`-хелпери:

```go
r.GetE("/users/{id}", func(w http.ResponseWriter, req *http.Request) error {
    u, err := load(mux.Param(req, "id"))
    if err != nil {
        return err
    }
    return resp.JSON(w, u)
})
```

Ненульовий `error` іде в `ErrorHandler` роутера. Якщо його не задано, дефолт
відповідає загальним JSON `500` через `goloop/resp` і не розкриває текст помилки.
`nil` означає, що хендлер уже написав відповідь - тоді нічого не відбувається.

Повертай помилку **до запису** статусу чи тіла. Якщо хендлер уже записав, а
потім повернув помилку, error-handler пропускається (інакше він зіпсував би
відповідь другим статусом і зайвими байтами), тож клієнт отримує ту часткову
відповідь, яку хендлер устиг написати.

Роутер треба створювати через `mux.New()`; нульове значення `Router{}` панікує з
ясним меседжем, а не nil-розіменуванням.

Доступно: `GetE`, `PostE`, `PutE`, `PatchE`, `DeleteE`, `MethodE`, `HandleError`.

## Монтування хендлерів

```go
r.Mount("/admin", adminHandler)      // обслуговує /admin і /admin/...
r.MountStrip("/admin", adminRouter)  // те саме, але спершу зрізає /admin
```

- `Mount` передає хендлеру оригінальний шлях. Для хендлерів, що очікують повний
  шлях.
- `MountStrip` застосовує `http.StripPrefix`, тож самодостатній хендлер чи
  суброутер маршрутизує відносно власного кореня.

Обидва реєструють точний префікс і піддерево окремо, тож редіректу з `/admin` на
`/admin/` немає.

## Власні 404 і 405

```go
r := mux.New(
    mux.WithNotFound(notFound),
    mux.WithMethodNotAllowed(methodNotAllowed),
)
```

При невідповідності методу заголовок `Allow`, обчислений стандартним mux,
копіюється у відповідь **перед** вашим хендлером. Якщо жодну опцію не задано,
використовуються стандартні `404`/`405`, і на збіжному шляху немає жодного
додаткового навантаження.

## Конкурентність

Реєструйте всі маршрути до старту сервера. Як і `http.ServeMux`, `Router`
безпечний для конкурентного обслуговування після того, як маршрути на місці;
реєстрація маршрутів паралельно з обслуговуванням не підтримується.
